HTTP Form-based authentication and DNS

HTTP form-based authentication was created to mitigate the downsides of the basic authentication. Form-based authentication uses the standard HTML form fields to pass the user name and password values to the server. It does not send a GET request with the credentials. Instead, it sends a POST request and passes the credentials in the post body.

First, going into a site: testing-ground.webscraping.pro/login

It is not a secure site as it uses HTTP.

To capture the traffic that will happen Wireshark is opened and entered the network interface (Ethernet 3) with a capture filter port 80 for capturing HTTP traffic.

Filled in the values in the HTTP form with the right credentials and clicked on “login”. After getting the welcome message, we can now come back to the Wireshark window and stop capturing.

Upon clicking on POST packet sent to the server, the username and the password being sent in clear text to the server can be easily seen under the HTML Form URL Encoded expansion. It is not even base 64 encoded. So it is not doing obfuscation. Using form-based authentication with HTTP is even less secure than the basic authentication method. In the case of HTTPS, since the entire packet is encrypted anyway, there is no need to worry about any encoding or obfuscation. The form-based authentication was created keeping HTTPS in mind.

Now to capture DNS traffic we use port 53 as our capture filter and start capturing packets from the network interface (Ethernet 3). To generate some DNS traffic we can ping www.domain.com in the Windows command prompt. After capturing some traffic the capture was stopped.

It is visible from the captures above, that DNS traffic is not encrypted at all. We can see the host domain name, which server it was pinged to and we can even see the IP address of the host from the info section itself.
