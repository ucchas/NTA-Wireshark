Telnet only needs to send keystrokes to the remote server in real time. Although real time applications are the job of UDP, telnet sends the data over TCP because telnet traffic must be sent reliably. TCP uses a cool trick here to send the traffic in a near real-time fashion.

Upon inspection of the first TELNET packet, under the expansion Transmission Control Protocol > Flags, here it is observed that the push flag is set to 1. This means whenever any character comes from the telnet, just send it. Don’t wait as the TCP does to combine a whole PDU (Protocol Data Unit) and then send it.

By doing right-click > Follow > TCP Stream:

Here on the new window, everything is visible that we saw previously from our telnet session. This, in turn, means that Telnet is not encrypted at all and it is not secure to use it for communication with our remote devices.

Also, in the TCP Stream data shown above, the letters of the username are shown twice. That is because, in Wireshark, the red color represents what we sent to the server and the blue represents what the server sends back to us. What telnet does is it echos back to us everything that we have to see back on our screen. But while typing in the password, the server does not send the data back to us. While logging into telnet also the password string is not shown on the command-line screen. That’s why the password is shown in all red colors on the TCP Stream data acquired by Wireshark.
